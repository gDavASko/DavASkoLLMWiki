A Layered, Self-Validating Knowledge Base with Hybrid Retrieval for LLM Agents: Method and Empirical Evaluation on a Real Corpus

DavASko  ·  engineering & analysis assisted by Claude (Anthropic)
DavASko LLM Wiki framework · v3.x · Report compiled 2026-06-23 · Offline, reproducible

Abstract

We present DavASko LLM Wiki, a multi-layered, Obsidian-compatible knowledge-base framework with a built-in, fully offline hybrid retrieval engine (exact symbolic matching + dense semantic search over Jina v3 embeddings) intended to ground software-engineering LLM agents. The central, often-unanswered question for such systems is whether a curated retrieval layer actually outperforms simply letting the agent grep and read the source. We answer it empirically. On a real, deployed 162-document corpus (the KBPro knowledge base) with 15 labeled questions, the semantic retriever attains recall@5 = 0.633 and MRR = 0.718, versus a lexical (grep) baseline of 0.333 / 0.435 — roughly doubling recall and improving first-relevant ranking by 65%. We further report controlled ablations: a measurement-driven ranking fix raised the hybrid ranker's MRR from 0.641 to 0.718 (+12%); structure-aware chunking beat fixed-window chunking by +7.8% MRR at equal recall; and batched embedding, proven numerically identical to single-item embedding (cosine ≥ 0.99999997), yields a modest ~11% CPU speedup. We release the evaluation harness, 32 unit tests, and configuration so every number here is reproducible, and we document the method's limitations candidly.

1. Introduction

LLM coding agents increasingly operate against large, evolving project knowledge: design docs, architecture notes, runbooks, and source code. A common pattern is Retrieval-Augmented Generation (RAG) [1]: retrieve relevant context, then condition the model on it. Yet a foundational question is frequently asserted rather than tested: does a curated, indexed knowledge layer beat the trivial baseline of letting the agent search the files directly (ripgrep + read)? A retrieval layer that does not beat grep is negative-value complexity — it adds maintenance burden and a second, drifting copy of the truth.

This report describes the DavASko LLM Wiki framework and, more importantly, subjects its retrieval engine to measurement on a real corpus with a labeled query set, a grep baseline, and standard information-retrieval metrics. Our contributions are: (i) a fully-offline hybrid retrieval design with several engineering choices (IVF-style multi-probe, an adaptive per-query threshold, structure-aware chunking, ground-truth-preferring ranking, and content-hash provenance for drift detection); (ii) an evaluation harness reporting recall@k / MRR / nDCG@k against flat-cosine and lexical baselines; and (iii) an empirical study on the deployed KBPro knowledge base, including controlled ablations that changed the design based on observed data.

2. Background & Related Work

Dense retrieval & RAG. Dense passage retrieval embeds queries and passages into a shared vector space and ranks by cosine similarity [2]; RAG couples this with a generator [1]. We use Jina Embeddings v3 [3], a multilingual model with task-specific LoRA adapters and asymmetric query:/passage: prefixes, run locally via ONNX Runtime in Transformers.js [4].

Approximate nearest neighbour. Inverted-file (IVF) indexes partition vectors into clusters and probe the nprobe nearest at query time to trade recall for speed [5]. We adopt the same nprobe control but, given small corpora, default to exhaustive search (Section 4).

Chunking. Fixed-size sliding windows are the common default; structure-aware / recursive splitters cut on document structure (headings, paragraphs) and are widely reported to improve coherence [6]. We test both head-to-head.

Metrics. We use recall@k, Mean Reciprocal Rank (MRR), and normalized Discounted Cumulative Gain (nDCG@k), the standard ranking metrics [7, 8]. Hybrid lexical+dense retrieval is established practice [9]; our symbolic stream is an exact-identifier matcher specialized for code symbols.

3. System Architecture

Knowledge is organized into independent layers (e.g. llm-wiki, framework-wiki, project layers) with a strictly downward dependency graph. Each layer separates immutable source snapshots (raw/) from curated, AI-maintained pages (wiki/). The retrieval engine indexes both. Figure 1 shows the offline indexing and online query pipelines.

4. Retrieval Method

4.1 Hybrid two-stream search

Stream A (symbolic) matches strict code identifiers — PascalCase with ≥2 humps, I-interfaces, m_ fields — against document id, symbols, tags and wiki-links. Matches are scored by field type (id > symbols > tags > wiki-links) and down-weighted by a mini-IDF (1/√df) so a common identifier cannot flood the top-k. Stream B (semantic) embeds the query with the query: prefix and scores chunks by cosine. The two streams are merged by a single effective score (Section 7.1), not by a hard "symbols first" tier.

4.2 Multi-probe routing

Vectors are stored one shard per layer; each shard's centroid is the mean of its member vectors. At query time the engine ranks shards by centroid similarity and scans the nprobe nearest (IVF multi-probe). When nprobe ≥ the number of shards — the common small-corpus case — search is exhaustive and incurs no recall loss.

4.3 Adaptive threshold

A fixed cosine cutoff is fragile: score distributions shift with model, language and length, so no universal value exists. The default relative mode sets a per-query threshold τq = max(junk_floor, α·maxd sim(q,d)) (defaults α=0.85, junk_floor=0.35), adapting to each query while flooring out noise. A legacy absolute mode remains for calibrated deployments.

4.4 Structure-aware chunking

The indexer tokenizes Markdown into headings, paragraphs and atomic code blocks, packs them to a target size within [min,max] words, hard-splits only oversized blocks, merges undersized fragments, and prepends a heading breadcrumb (e.g. [Event Bus > API]) so each chunk carries its section context. This avoids mid-sentence cuts, split code, and topic-mixed embeddings inherent to fixed windows.

4.5 Ground-truth ranking & provenance

Code/raw/ is the source of truth; wiki/ pages are derived summaries that can lag. On near-ties the engine ranks the source above the summary and labels results SOURCE vs SUMMARY. Each page records content-hash provenance of its cited sources; a validator flags pages whose sources changed (detectable drift), feeding an AI re-actualization workflow rather than silent staleness.

7. Ablation Studies

7.1 Fixing the hybrid ranker (measurement-driven)

The first real-corpus run exposed a defect: the original merge placed all symbol matches above semantic ones, so tangential exact matches outranked the best semantic hits — the hybrid ranked worse than pure semantic (MRR 0.641 vs 0.718). Two changes, each measured on the same index, closed the gap:

baseline — strict "symbols-first" tiers0.6410.577

7.3 Indexing speed: GPU, batching, and proven parity

Indexing embedded one chunk per model call on CPU. We addressed speed on two fronts, each gated by a correctness check first.

Batched embedding. We batch a document's chunks into one model call. Correctness was proved before measuring speed: across mixed-length samples in one batch, batched vs single-item embeddings are numerically identical — minimum cosine 0.99999997, max absolute element difference 2.8×10-5 (FP noise). On CPU the speed-up is modest — 114 s → 101 s (~11%) for 45 chunks — because a 570M-parameter forward pass saturates CPU compute; batching only removes per-call overhead.

GPU acceleration (the real lever). The vendored ONNX runtime ships a DirectML execution provider (any DX12 GPU: NVIDIA/AMD/Intel). The engine auto-detects the GPU and falls back to CPU, logging the device used. Measured on 12 embeddings: CPU 9657 ms vs GPU 1205 ms — an 8.0× speed-up, with min cosine(CPU, GPU) = 0.999984 (numerically equivalent for retrieval). Full-corpus indexing drops from ~30 min on CPU to a few minutes on GPU. This dwarfs batching and removes the indexing bottleneck where a GPU is present.

8. Discussion

The value question is answered — on this corpus. The dense layer is not decorative: it nearly doubles recall and markedly improves ranking over grep. For natural-language questions, semantic search is the workhorse; the symbolic stream is neutral here but is designed to add precision on identifier-style queries (e.g. CowController) that this question set does not contain.

Measurement changed the design. Every improvement in Section 7 was prompted by data, not intuition: the ranking defect was invisible until measured, and its root cause (one acronym) was found by per-query inspection. This is the intended operating mode — the harness is part of the engineering loop, not a one-off.

Clustering is currently irrelevant. With two shards, multi-probe equals exhaustive search; the architecture's IVF machinery only begins to matter at many layers / large corpora, where nprobe must be calibrated (the harness supports this).

9. Limitations & Threats to Validity

  • Small query set (n=15). Single-query effects move aggregate metrics; confidence intervals are wide. Numbers indicate direction, not precision. Expanding the labeled set is the top priority for stronger claims.
  • Single corpus / domain. Results are from one deployed knowledge base; generalization to other corpora is untested.
  • Label proxy. "Required sources" annotations approximate relevance; they were authored for a Q&A regression set, not as exhaustive relevance judgments, so recall may be under- or over-counted.
  • Partial index for the eval. The reported table indexes 2 of 6 layers (those covering the labels); fewer distractor documents can make retrieval optically easier than full-corpus deployment.
  • Recall ceiling. recall@5 = 0.633 means ~37% of relevant material is missed in the top-5; a re-ranker would not fix recall (it reorders), so the next lever is larger k, query expansion, or chunk/representation changes — to be decided by measurement.
  • Indexing throughput. CPU indexing is slow (~2.3 s/chunk); GPU (DirectML) removes this (~8×) where a DX12 GPU and a GPU-enabled ONNX runtime are present, otherwise the system falls back to CPU.
  • Drift remains a process, not a guarantee. Provenance hashing makes staleness detectable; it does not eliminate the dual-source-of-truth between code and wiki.

10. Reproducibility

All artifacts are offline and version-controlled. The retrieval core has 32 unit tests; the embedding parity check gates batching. To reproduce the tables:

# build the index (offline; model + deps vendored)
node system/build-index.js --force
# main results + baselines (Table 1, Figure 2)
node system/scripts/eval-retrieval.js
# threshold/alpha calibration sweep
node system/scripts/eval-retrieval.js --sweep
# chunking A/B (Table 3): flip chunk_strategy in system/index-config.json, rebuild, re-eval
# batching parity (Section 7.3)
node system/scripts/check-embed-parity.mjs
# unit tests of the scoring core
npm test

Key configuration (system/search-config.json, system/index-config.json): threshold_mode=relative, α=0.85, junk_floor=0.35, top_k=5, nprobe=8, ground_truth_boost=0.05; chunk_strategy=structural, target 250 / min 80 / max 400 words, index_code=true, embed_batch_size=16.

11. Conclusion

We turned a frequently-asserted claim into a measured one: on a real, deployed knowledge base, a layered hybrid retrieval engine substantially outperforms the grep baseline (recall@5 0.633 vs 0.333, MRR 0.718 vs 0.435). Beyond the headline, the study demonstrates a disciplined, measurement-driven engineering loop — two ranking fixes (+12% MRR) and a chunking choice (+7.8% MRR) were adopted only after the data justified them, and a speed optimization was shipped only after numerical parity was proven. The remaining gaps (small query set, recall ceiling, CPU indexing) are documented rather than hidden, and each maps to a concrete, falsifiable next experiment.
